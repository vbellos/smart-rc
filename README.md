# Smart RC

ESP32-S3 firmware for a small RC platform driven by a TB6612FNG dual H-bridge.
A rear DC motor handles forward/reverse; a front DC motor (no servo, no
position feedback) handles steering via **open-loop timed pulses**. Control is
exposed over a Wi-Fi web portal and a small JSON API.

Target board: **4D Systems gen4-ESP32-S3 R8N16** (8 MB octal PSRAM, 16 MB flash).

---

## Wiring

| TB6612 pin | Connect to              |
|------------|-------------------------|
| `VCC`      | ESP32 `3V3` (logic)     |
| `STBY`     | ESP32 `3V3` (always-on) |
| `GND`      | ESP32 `GND`             |
| `VM`       | External motor supply   |
| `PWMA`     | GPIO **4**              |
| `AIN1`     | GPIO **5**              |
| `AIN2`     | GPIO **6**              |
| `PWMB`     | GPIO **7**              |
| `BIN1`     | GPIO **8**              |
| `BIN2`     | GPIO **9**              |
| `A01/A02`  | Front **steering** motor |
| `B01/B02`  | Rear **drive** motor     |

Pin assignments are in `src/Pins.h`. PWM is generated by the ESP32 LEDC at
20 kHz, 8-bit (0â€“255).

> âš ď¸Ź Tying `STBY` permanently high disables the TB6612's standby feature.
> If you later want to power-cycle the bridge from firmware, free a GPIO and
> add it to `Pins.h` + `MotorDriver::begin()`.

## Boot behavior

1. **Motors are initialised first** and forced to a stopped state before any
   networking or config code runs.
2. Persistent config is loaded from NVS (`Preferences` namespace `smartrc`).
3. The drive/steering layers and safety watchdog are configured from that
   config.
4. The network manager attempts STA mode with the saved Wi-Fi credentials.
   - If `useDhcp=false`, the saved static IP/gateway/subnet/DNS is applied.
   - If the join fails (or no SSID is configured), the device falls back to
     an AP hotspot using `apSsid` / `apPassword` (default
     `SmartRC-Setup` / `smart1234`).
5. The web portal starts on port 80.

After boot, the main loop services the network manager (auto-reconnect),
the web server, the steering state machine, the safety watchdog, and the
sensor stub.

### Safety guarantees

- Motors are stopped on boot.
- A heartbeat tracks every accepted command. If the gap exceeds
  `heartbeatTimeoutMs` (default 800 ms) **both motors are stopped**.
- Steering reversal is held off for `steeringCooldownMs` (default 120 ms)
  after the previous pulse to avoid mechanical shock and current spikes.
- `POST /api/control { "action": "estop" }` latches both motors off until
  `clear_estop` is sent.

## HTTP / JSON API

All endpoints accept and return JSON. The portal HTML is served from `/`.

### `GET /api/status`

```json
{
  "uptime_ms": 12345,
  "heap_free": 187456,
  "net":   { "mode": "sta", "ip": "192.168.1.42", "ssid": "Home",
             "rssi": -57, "hostname": "smartrc", "connected": true },
  "motors":{ "drive_moving": false, "steer_state": 0, "steer_last_dir": 0 },
  "safety":{ "emergency": false, "stale": false }
}
```

`steer_state`: 0=Idle, 1=Pulsing, 2=Cooldown.
`steer_last_dir`: 0=None, 1=Left, 2=Right.

### `GET /api/config`

Returns every persisted setting (Wi-Fi creds, AP creds, hostname, IP mode,
static IP fields, steering pulse/cooldown, default PWMs, heartbeat timeout).

### `POST /api/config`

Body: a partial JSON object with any of the keys returned by `GET`.
Saves to NVS. Tunables that don't require restart (steering pulse, default
PWMs, heartbeat) are applied live; Wi-Fi/IP changes take effect after a
reboot.

### `POST /api/control`

```json
{ "action": "forward", "speed": 200 }
```

| `action`      | Effect                                                     |
|---------------|------------------------------------------------------------|
| `forward`     | Drive motor forward                                        |
| `reverse`     | Drive motor reverse                                        |
| `stop`        | Drive motor coast                                          |
| `brake`       | Drive motor short-brake                                    |
| `left`        | Start a left steering pulse                                |
| `right`       | Start a right steering pulse                               |
| `estop`       | Latch emergency stop                                       |
| `clear_estop` | Release emergency stop                                     |

`speed` is optional (0â€“255). When omitted or `0`, the relevant
"default PWM" from config is used. A rejected steering pulse (cooldown not
elapsed) returns HTTP 409 with `{ "ok": false, "reason": "steering busy" }`.

### `GET /api/wifi/scan`

Triggers a synchronous Wi-Fi scan (takes ~2â€“5 s). If the radio is currently
AP-only it is temporarily promoted to `AP+STA` so the portal session stays
alive. Returns:

```json
{
  "count": 4,
  "networks": [
    { "ssid": "Home",    "rssi": -48, "secure": true,  "channel": 6 },
    { "ssid": "GuestAP", "rssi": -71, "secure": false, "channel": 11 }
  ]
}
```

### `POST /api/wifi/provision`

**Scan â†’ pick â†’ try â†’ save** in one call. Body:

```json
{ "ssid": "Home", "password": "hunter2" }
```

The device keeps its AP up (if running), switches to `AP+STA`, and attempts
to join the given network with a 12 s timeout. On success it writes the
credentials to NVS and returns:

```json
{ "ok": true, "saved": true, "ssid": "Home",
  "ip": "192.168.1.42", "rssi": -49, "message": "connected and saved" }
```

On failure it rolls back to the previous mode (so the hotspot stays up)
and returns HTTP 409:

```json
{ "ok": false, "reason": "auth failed" }   // or "ssid not found" / "timeout"
```

Nothing is saved unless the connection actually succeeds â€” which means
you can't brick provisioning by entering a wrong password.

### Reset / reboot

| Endpoint                | Effect                                          |
|-------------------------|-------------------------------------------------|
| `POST /api/reboot`      | Soft reset                                      |
| `POST /api/reset/network` | Wipe Wi-Fi/IP only â€” keeps motor + AP config |
| `POST /api/reset/factory` | Wipe everything and reboot                   |

## Development

```sh
pio run                              # build firmware
pio run -t upload                    # flash device
pio device monitor                   # serial console
pio run -t upload -t monitor         # all-in-one

pio test -e native_test              # host-side unit tests (steering FSM)
```

The native test environment uses Unity and a stub `Arduino.h` under
`test/native_stubs/` to exercise `Steering` without an ESP32. To add a
new host-testable module, drop its `.cpp` into `build_src_filter` in
`platformio.ini`.

For on-device tests, create `test/test_<name>/test_<name>.cpp` and run
`pio test` (the device env runs Unity over the serial link). The native
test directory is excluded from the device build via `test_ignore`.

## Module layout

```
src/
  main.cpp                  // boot / wiring of all subsystems
  Pins.h                    // fixed GPIO + PWM constants
  config/        Config.{h,cpp}       // NVS-backed config model + reset
  network/       NetworkManager.{h,cpp} // STA, AP fallback, reconnect
  portal/        Portal.{h,cpp}, PortalHtml.h  // web server + UI
  motors/        MotorDriver.{h,cpp}  // TB6612 LEDC abstraction
                 Drive.{h,cpp}         // rear motor: fwd/rev/stop
                 Steering.{h,cpp}      // front motor: timed pulses + cooldown
  control/       CommandHandler.{h,cpp} // single command funnel
                 Safety.{h,cpp}         // heartbeat watchdog + e-stop
  sensors/       Sensors.{h,cpp}        // intentional empty stub
test/
  native_stubs/  Arduino.h              // host-side stub for native tests
  test_steering_native/                 // Unity tests for Steering FSM
```

`MotorDriver` knows nothing about "drive" vs "steering" â€” it only takes
`(channel, dir, speed)`. `Drive` and `Steering` are thin policy layers on top.
This keeps the TB6612 abstraction reusable if you later add a second driver
chip or swap motors around.

## Future expansion

- **Sensors.** `src/sensors/` is a deliberate empty extension point. A
  suggested API is sketched in `Sensors.h` â€” add `update()` work, then
  call `sensors::appendStatusJson(...)` from `Portal::handleStatus()` so
  readings show up in `/api/status` automatically.
- **Lower-latency control.** The current portal uses synchronous
  `WebServer.h` for simplicity. For sub-50 ms control, replace with
  `ESPAsyncWebServer` + a WebSocket channel â€” `CommandHandler::execute()`
  is already the single chokepoint to call from the new transport.
- **OTA.** `ArduinoOTA` or the ESP32 HTTP-update library both drop in
  cleanly: register them in `setup()` after `g_net.begin()` and pump them
  from `loop()`.
- **Steering closed loop.** When/if a feedback sensor (encoder, pot, IMU
  yaw) is added, build a new `SteeringClosedLoop` policy beside `Steering`
  and have `CommandHandler` pick which one to call. The TB6612 layer
  doesn't need to change.
- **Persistent telemetry.** The NVS namespace is `smartrc`; add new keys
  in `Config.{h,cpp}` and they'll round-trip through `GET/POST /api/config`
  with one extra line each.
